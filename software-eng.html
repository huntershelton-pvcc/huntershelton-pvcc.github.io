Chapter 7: Software Engineering

Hunter Shelton

Item Definition
software engineering the branch of computer science that seeks principles to guide the development of large, complex software systems.
integrated development environment (IDE) systems that combine tools for developing software (editors, compilers, debugging tools, and so on) into a single integrated package.
Association for Computing Machinery (ACM)
  1. purpose of ACM
  2. benefits of ACM membership
  3. student ACM membership

  1. an international scientific and educational organization dedicated to advancing the arts, sciences, and applications of information technology.
  2. free subscription to Communications; online resources through the ACM Learning Center, and the ability to participate in ACM Distinguished Speakers Program
  3. program for students studying computing at an accredited institution
Institute of Electrical and Electronics Engineers (IEEE)
  1. purpose of IEEE
  2. benefits of IEEE membership
  3. student IEEE membership
  1. to foster technological innovation and excellence for the benefit of humanity
  2. free subscription to Communications; online resources through the ACM Learning Center, and the ability to participate in ACM Distinguished Speakers Program
  3. a student member must carry at least 50% of a normal full-time academic program as a registered undergraduate or graduate student in a regular course of study in IEEE designated fields
Software Life Cycle: Development Phase
  1. Requirements analysis
  2. Design
  3. Implementation
  4. Testing
  1. specify what services the proposed system will provide, to identify conditions on those services, and define how the outside world will interact with the system
  2. design involves creating a plan for the construction of the proposed system
  3. involves the actual writing of programs, creation of data files, and development of databases
  4. the process of debugging programs and confirming that the final software product was compatible with the software requirements specifications
modular implementation the division of software into manageable units, generically called modules, each of which deals with only a part of the software's overall responsibility
Unified Modeling Language (UML) collection of tools that has been developed with the object-oriented paradigm in mind
use case interactions between the system and its users
use case diagram depicts the proposed system as a large rectangle in which interactions between the system and its users are represented as ovals and users of the system are represented as stick figures.
Data relationships:
  1. one-to-one
  2. one-to-many
  3. many-to-many
  1. the association between patients and occupied private rooms in that each patient is associated with only one room and each room is associated with only one patient
  2. the association between physicians and patients in that one physician is associated with many patients is associated with one (primary) physician
  3. would occur if we included consulting physicians in the physician-patient relationship. Then each physician could be associated with many patients and each patient could be associated with many physicians
design pattern predeveloped model for solving a recurring problem in software design
Software quality assurance (SQA) groups which are charged with overseeing and enforcing the quality control systems adopted by the organization
Software testing:
  1. glass box testing
  2. black box testing
  3. beta testing
  4. alpha testing
  1. the software tester is aware of the interior structure of the software and uses this knowledge when designing the test
  2. tests that do not rely on knowledge of the software's interior composition
  3. a preliminary version of the software is given to a segment of the intended audience with the goal of learning how the software performs in real-life situations before the final version of the product is solidified and released to the market
  4. similar to beta testing, but performed at the developer's site
Documentation:
  1. user documentation
  2. system documentation
  3. technical documentation
  1. explain the features of the software and describe how to use them
  2. describe the software's internal composition so that the software can be maintained later in its life cycle
  3. describe how a software system should be installed and serviced
Intellectual property law
  1. software license
  2. software disclaimer
  3. best defenses against software liability claims
  1. a legal agreement between the owner and user of a software product that grants the user certain permissions to use the product without transferring ownership rights to the intellectual property
  2. disclaimers in the software licenses that state the limitations of their liability
  3. apply sound software engineering principles during the software's development, use a level of care compatible with the software's application, and produce and maintain records that validate these endeavors
